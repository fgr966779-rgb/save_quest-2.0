# PiggyVault

**PiggyVault** — це мобільний додаток для управління особистими фінансами, розроблений на базі Flutter. Додаток допомагає користувачам відстежувати витрати, планувати заощадження, управляти бюджетом та досягати фінансових цілей у зручний та інтуїтивний спосіб. Завдяки кросплатформній природі Flutter, PiggyVault працює на Android та iOS з єдиним кодовим базисом, забезпечуючи однаковий досвід на всіх пристроях.

## Огляд проєкту

PiggyVault створений для вирішення поширеної проблеми — відсутності простого та надійного інструменту для повсякденного управління фінансами. Багато існуючих рішень або занадто складні у використанні, або містять надмірну рекламу, або вимагають платної підписки для базових функцій. PiggyVault пропонує альтернативу: чистий, швидкий і функціональний додаток, який зосереджений на тому, що дійсно важливо користувачу.

Основна філософія проєкту полягає у тому, що управління фінансами має бути простим і приємним. Замість перевантаженого інтерфейсу з десятками непотрібних функцій, PiggyVault зосереджується на ключових можливостях: запис транзакцій, аналіз витрат, встановлення фінансових цілей та візуалізація прогресу. Кожен елемент інтерфейсу продуманий для мінімізації кількості кроків, необхідних для виконання дії.

Додаток ідеально підходить для різних типів користувачів: студентів, які хочуть контролювати витрати на навчання, сімей, що планують бюджет на місяць, або підприємців, яким потрібен швидкий облік доходів і витрат. Гнучка система категорій та тегів дозволяє адаптувати додаток під індивідуальні потреби кожного користувача.

### Візуальна ідентичність

Назва "PiggyVault" поєднує два символи: **Piggy** (скарбничка) — традиційний символ заощадження та фінансової свідомості, та **Vault** (сейф) — символ безпеки та надійності. Разом вони відображають основні принципи додатка: допомога у заощадженні та забезпечення безпеки фінансових даних користувача. Візуальний дизайн додатка використовує м'які пастельні кольори з яскравими акцентами, що створює приємну атмосферу та мотивує до фінансової дисципліни.

## Основний функціонал

### Облік транзакцій

Серцем PiggyVault є система швидкого та зручного запису фінансових транзакцій. Користувач може додати нову транзакцію за кілька секунд: вказати суму, обрати категорію, додати нотатку та встановити дату. Додаток підтримує як доходи, так і витрати, дозволяючи вести повний фінансовий облік в одному місці. Для зручності передбачена можливість створення повторюваних транзакцій, що автоматично додаються у визначені дати — ідеально для щомісячних платежів за оренду, підписками або регулярних надходжень.

Кожна транзакція може бути доповнена тегами для детальнішої класифікації. Наприклад, витрату в ресторані можна позначити тегами "вихідні" та "зустріч з друзями", що дозволить згодом аналізувати витрати за різними критеріями. Система підтримує як попередньо встановлені категорії (їжа, транспорт, розваги, житло тощо), так і можливість створення власних користувацьких категорій з індивідуальними іконками та кольорами.

### Аналіз та статистика

PiggyVault надає потужний інструментарій для аналізу фінансових даних. Інтерактивні діаграми та графіки дозволяють візуально оцінити розподіл витрат за категоріями, динаміку доходів та витрат у часі, а також порівняти фінансові показники за різні періоди. Користувач може переглядати звіти за день, тиждень, місяць чи рік, вибираючи найзручніший період для аналізу.

Особливістю аналітичного модуля є функція розпізнавання патернів витрат. Додаток автоматично виявляє тенденції, такі як збільшення витрат на певну категорію або сезонні коливання доходів, та сповіщає користувача про виявлені закономірності. Це допомагає приймати обґрунтовані фінансові рішення та вчасно коригувати бюджет.

### Фінансові цілі

Одна з ключових функцій PiggyVault — система керування фінансовими цілями. Користувач може встановити ціль (наприклад, "відпустка", "новий ноутбук", " emergency fund") та визначити цільову суму і термін досягнення. Додаток автоматично розраховує необхідний щомісячний внесок та відстежує прогрес у реальному часі.

Візуальний прогрес-бар та повідомлення-нагадування мотивують користувача дотримуватися плану заощаджень. Крім того, система дозволяє створювати кілька одночасних цілей з пріоритезацією, що допомагає розподіляти наявні ресурси найефективніше. Додаток також пропонує розумні поради щодо оптимізації бюджету на основі аналізу поточних витрат користувача.

### Безпека даних

PiggyVault ставить безпеку фінансових даних користувача на перше місце. Усі дані шифруються за допомогою алгоритму AES-256 перед збереженням на пристрої. Для додаткового захисту передбачена біометрична аутентифікація (відбиток пальця або розпізнавання обличчя), а також можливість встановлення PIN-коду. У разі втрати пристрою користувач може віддалено знищити всі фінансові дані через веб-панель.

Опціональна хмарна синхронізація забезпечує резервне копіювання даних та доступ до фінансової інформації з різних пристроїв. Синхронізація виконується через захищене з'єднання з end-to-end шифруванням, що гарантує конфіденційність даних навіть на стороні сервера. Користувач має повний контроль над тим, які дані синхронізуються та як часто виконується резервне копіювання.

## Архітектура проєкту

PiggyVault побудований за принципами чистої архітектури (Clean Architecture), що забезпечує чітке розділення відповідальності між шарами додатка та полегшує тестування і масштабування. Архітектура складається з трьох основних шарів:

### Шар представлення (Presentation Layer)

Цей шар відповідає за відображення даних та взаємодію з користувачем. Він реалізований за допомогою патерну BLoC (Business Logic Component), який відокремлює бізнес-логіку від інтерфейсу. Кожен екран додатка має власний BLoC, що управляє станом та обробляє події користувача. Шар представлення не містить жодної бізнес-логіки і виключно відображає дані, отримані від шару домену.

UI-компоненти організовані у вигляді повторно використовуваних віджетів, що дозволяє зберігати консистентний вигляд додатка на всіх екранах. Дизайн-система включає визначену палітру кольорів, типографіку, набір іконок та стандартні компоненти (кнопки, поля вводу,卡片 тощо), що значно прискорює розробку нових екранів та забезпечує єдиний стиль.

### Шар домену (Domain Layer)

Доменний шар містить основну бізнес-логіку проєкту. Тут визначаються сутності (Entity), об'єкти значень (Value Object) та контракти для репозиторіїв. Цей шар не залежить від жодної зовнішньої бібліотеки чи фреймворку, що робить його максимально незалежним та легким для тестування. Вся бізнес-логіка реалізована у вигляді Use Cases (випадків використання), кожен з яких інкапсулює одну конкретну бізнес-операцію.

Такий підхід дозволяє легко модифікувати бізнес-логіку без впливу на інтерфейс чи рівень доступу до даних. Наприклад, зміна алгоритму розрахунку бюджету не вимагає жодних змін у коді UI або бази даних.

### Шар даних (Data Layer)

Шар даних відповідає за отримання, збереження та кешування інформації. Він реалізує інтерфейси репозиторіїв, визначені на рівні домену, та забезпечує взаємодію з локальною базою даних (SQLite через Drift/ORM) та віддаленим API (якщо використовується хмарна синхронізація). Шар даних містить моделі DTO (Data Transfer Object) та логіку конвертації між DTO та сутностями домену.

Керування станом даних здійснюється через патерн Repository, який абстрагує джерело даних від споживача. Це означає, що шар домену та представлення не знають, звідки приходять дані — з локальної бази, мережі або кешу. Такий підхід дозволяє легко змінювати джерело даних або додавати нові (наприклад, перехід з SQLite на Hive або додавання REST API клієнта) без модифікації інших шарів.

## Технологічний стек

| Компонент | Технологія | Опис |
|---|---|---|

| **DI** | get_it + injectable | Внедрення залежностей |
| **Тестування** | mockito + integration_test | Юніт-тести, віджет-тести та інтеграційні тести |

## Встановлення та налаштування

### Клонування репозиторію

```bash
git clone https://github.com/your-username/piggyvault.git
cd piggyvault
```

### Встановлення залежностей

```bash
flutter pub get
```

### Генерація коду

PiggyVault використовує code generation для ряду компонентів (Drift таблиці, DI реєстратори тощо):

```bash
dart run build_runner build --delete-conflicting-outputs
```

### Налаштування змінних середовища

Створіть файл `.env` у кореневій директорії проєкту на основі прикладу:

```bash
cp .env.example .env
```

Відредагуйте `.env`, вказавши необхідні значення для Firebase конфігурації та інших сервісів.

### Запуск у режимі розробки

```bash
# Запуск на підключеному пристрої або емуляторі
flutter run

# Запуск у режимі профілювання
flutter run --profile

# Запуск з вибором конкретного пристрою
flutter run -d <device_id>
```

### Запуск з використанням FVM

Якщо проєкт використовує FVM (Flutter Version Management):

```bash
fvm use
fvm flutter pub get
fvm flutter run
```

## Розробка та внесок

Ми вітаємо внески від спільноти! Будь-хто може допомогти покращити PiggyVault, незалежно від рівня досвіду. Нижче наведено базові рекомендації для контриб'юторів.

### Гілкова стратегія

Проєкт використовує Git Flow з деякими адаптаціями:

- `main` — стабільна гілка, містить лише перевірені та готові до релізу зміни
- `develop` — робоча гілка, де інтегруються всі нові функції
- `feature/<name>` — гілки для розробки нових функцій
- `bugfix/<name>` — гілки для виправлення помилок
- `hotfix/<name>` — термінові виправлення на основі `main`

### Процес внеску

1. Форкніть репозиторій та створіть гілку `feature/your-feature-name`
2. Впровадьте зміни, дотримуючись принципів Clean Architecture
3. Додайте юніт-тести для нової функціональності
4. Переконайтеся, що всі існуючі тести проходять успішно
5. Оновіть документацію (код-коментарі, README при необхідності)
6. Створіть Pull Request з детальним описом змін

### Стандарт кодування

- Дотримуйтесь [Official Dart Style Guide](https://dart.dev/guides/language/effective-dart/style)
- Використовуйте `dart format` та `dart analyze` перед комітом
- Назви файлів використовують `snake_case` (наприклад, `transaction_repository.dart`)
- Назви класів використовують `PascalCase`
- Приватні membri починаються з `_`
- Максимальна довжина рядка — 80 символів
- Кожен файл повинен мати header-коментар з описом призначення

## Тестування

PiggyVault має багаторівневу стратегію тестування, що забезпечує високу якість та надійність коду.

### Юніт-тести

Юніт-тести перевіряють окремі класи та функції в ізоляції. Вони зосереджені на бізнес-логіці (Use Cases, Entities) та утилітарних функціях. Для створення моків використовується бібліотека `mockito`.

```bash
# Запуск всіх юніт-тестів
flutter test

# Запуск тестів з покриттям
flutter test --coverage

# Запуск тестів конкретного файлу
flutter test test/features/transactions/domain/usecases/
```

### Віджет-тести

Віджет-тести перевіряють поведінку окремих UI-компонентів. Вони верифікують, що віджети правильно відображають дані та реагують на взаємодію користувача.

```bash
# Запуск віджет-тестів
flutter test test/features/transactions/presentation/widgets/
```

### Інтеграційні тести

Інтеграційні тести перевіряють роботу всього додатка в цілому, імітуючи реальні сценарії використання. Вони запускаються на реальному пристрої або емуляторі.

```bash
# Запуск інтеграційних тестів
flutter test integration_test/
```

### Покриття коду

Проєкт має на меті досягти мінімального покриття коду на рівні 80% для бізнес-логіки та 70% для шару представлення. Звіти про покриття генеруються автоматично при кожному Pull Request через CI/CD пайплайн.

## Збірка та реліз

### Android (APK)

```bash
flutter build apk --release
```

Згенерований файл знаходиться за шляхом `build/app/outputs/flutter-apk/app-release.apk`.

### Android (App Bundle)

```bash
flutter build appbundle --release
```

Рекомендований формат для публікації в Google Play Store. Файл: `build/app/outputs/bundle/release/app-release.aab`.

### iOS

```bash
flutter build ios --release
```

Після збірки відкрийте Xcode та виконайте архівування та розподіл через App Store Connect.

### Версіонування

PiggyVault використовує семантичне версіонування (SemVer): `MAJOR.MINOR.PATCH`. Номер версії визначається у файлі `pubspec.yaml` та оновлюється вручну перед кожним релізом. У майбутньому планується автоматизація цього процесу через CI/CD.
