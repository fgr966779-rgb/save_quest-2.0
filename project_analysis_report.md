# Отчёт по глубокому анализу проекта PiggyVault (Обновление: Июнь 2026)

## 1. Обзор архитектуры
Проект успешно прошел этап MVP (Фаза 1). Архитектура базируется на:
*   **State Management:** Riverpod. Наблюдается тенденция к усложнению логики внутри `SavingsNotifier` (God Object), что требует декомпозиции.
*   **Database:** Drift (SQLite). Текущая схема жестко ориентирована на локальное использование и имеет ограничения масштабируемости.
*   **AI:** Интеграция с OpenRouter (DeepSeek). Промпты требуют оптимизации для повышения стабильности JSON-ответов.

## 2. Текущее состояние функций
Реализован базовый цикл: Депозит -> XP/Streak -> Награды.
*   **Завершено:** Система уровней, Skill Tree (Hacker/Magnate/Resilience), Терминал (CLI), базовый UI.
*   **В процессе:** Squads и Joint Goals (реализован UI и локальные таблицы, но отсутствует сетевой слой).

## 3. Анализ проблем и технического долга

### 3.1 Зависимости (Выполнено ✅)
Проблема с версией `intl` и Flutter SDK решена.

### 3.2 Дублирование и унификация (Актуально ⚠️)
*   **Проблема:** Системы `Achievements` и `Badges` всё еще разделены.
*   **Рекомендация:** Перейти к единой системе `Reward`, где тип награды (Badge, Skin, XP) определяется полем в БД.

### 3.3 Масштабируемость целей (Критично 🚨)
*   **Проблема:** В `database.dart` и `SavingsNotifier` логика завязана на `goal_a_amount` и `goal_b_amount`. Это блокирует создание 3-й и более целей.
*   **Решение:** Рефакторинг таблицы `Deposits` для связи Many-to-Many или One-to-Many с таблицей `Goals` через промежуточную таблицу `DepositAllocations`.

### 3.4 Качество кода (Улучшено 📈)
Количество предупреждений линтера снижено с 300+ до 11. Основные оставшиеся замечания касаются `use_build_context_synchronously` и неиспользуемых переменных.

### 3.5 Архитектура данных для Cloud Sync (Актуально ⚠️)
*   **Проблема:** Использование автоинкрементных ID в некоторых таблицах (например, `UserProfiles`) создаст конфликты при слиянии данных с разных устройств.
*   **Рекомендация:** Перейти на UUID для всех первичных ключей.

## 4. Phase 2 Readiness Assessment

### 4.1 Готовность к Социальным функциям (Squads)
Текущие модели `Squads` и `JointGoals` хранятся локально. Для Фазы 2 требуется:
1.  Вынос логики обновлений в Cloud Functions или использование Real-time DB (Firebase/Supabase).
2.  Внедрение `owner_id` для записей, чтобы разграничить права доступа.

### 4.2 Готовность к Cloud Sync (Firebase/CRDT)
Приложение **частично готово**.
*   **Плюсы:** Большинство таблиц уже используют UUID.
*   **Минусы:** Отсутствует поле `updated_at` и `server_version` для реализации эффективного мерджа (Last Write Wins или дельта-обновления). Схема с `goal_a/goal_b` несовместима с динамическим распределением средств в группе.

### 4.3 AI Промпты
Промпты в `BountyProvider` и `PriceAnalysisService` используют текстовую вставку "Respond ONLY with JSON". Это нестабильно.
*   **Рекомендация:** Использовать структурированный вывод (JsonSchema), если поддерживается провайдером, или усилить валидацию на стороне Dart.

## 5. Рекомендации по рефакторингу для Фазы 2

1.  **N-Goals Refactoring:**
    *   Создать таблицу `DepositAllocations(deposit_id, goal_id, amount)`.
    *   Удалить поля `goal_a_amount` и `goal_b_amount` из `Deposits`.
2.  **Sync Strategy (CRDT Lite):**
    *   Добавить колонку `is_synced` (bool) и `last_modified` (DateTime) во все таблицы.
    *   Приложение должно работать в режиме "Offline First", записывая изменения локально и помечая их для синхронизации.
3.  **SavingsNotifier Decomposition:**
    *   Вынести логику XP и Стриков в отдельные `Domain Services`, чтобы `SavingsNotifier` отвечал только за координацию транзакции.
4.  **AI Optimization:**
    *   Внедрить "Chain of Thought" в промпты для анализа финансового здоровья.
